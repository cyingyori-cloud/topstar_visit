# 一、项目背景与目标


## 1.1 为什么做这个产品


拓斯达当前销售经营中，拜访仍然是推进客户关系、识别真实需求、推动商机转化的核心动作。但在一线执行过程中，拜访工作存在四类典型问题：


第一，拜访准备高度依赖个人经验。新人不知道见客户前该准备什么，老销售虽有经验，但打法难以复制，团队很难形成统一标准。


第二，客户分层管理难落地。S 级客户、A 级客户、B/C 级客户在拜访频率、经营目标、投入方式上本应不同，但在实际管理中往往混在一起，导致高价值客户投入不足，低价值客户又消耗过多时间。


第三，拜访质量难评估。完成一次拜访不等于完成一次有效拜访。有没有见到关键人、有没有推进商机、有没有拿到下一步承诺，通常缺少统一、结构化、可量化的判断标准。


第四，管理层难以穿透一线过程。总监和区域经理能看到结果，但不一定能看清过程，不容易及时发现“哪些客户该见没见”“哪些拜访做了但无效”“哪些商机需要立即协同”。


因此，这套系统的核心目标不是再做一个“记录工具”，而是把拜访从“经验驱动”升级为“规则驱动 + AI 辅助 + 过程可视”的经营系统。


一句话概括：用 AI 把销售拜访从“靠人记、靠人想、靠人带”升级为“有标准、有助手、有闭环”的数字化作战体系。


## 1.2 当前系统聚焦的业务定义


当前版本系统聚焦的是“销售拜访经营”而非完整 CRM 全流程，主要覆盖四条主线：


| 主线 | 当前定义 | 关注重点 |

|------|----------|----------|

| 客户盘点 | 用客户清单、等级、行业、商机、联系人、拜访状态形成一线经营底盘 | 哪些客户重要，哪些客户久未触达 |

| 拜访推进 | 用任务、已完成拜访、覆盖率、待确认提醒推进日常执行 | 今天见谁，本周怎么排，哪些超期 |

| 拜访赋能 | 用知识库与 AI 教练输出案例、话术、方法论和准备模板 | 见客户前该怎么准备、怎么说 |

| 经营复盘 | 用商机分布、覆盖率、超期分析、Top 客户和洞察卡做管理复盘 | 团队投入是否有效，风险在哪里 |


当前系统尚未进入“真实订单、真实审批、真实工单、真实财务”阶段，因此本版本的核心价值在于建立拜访经营框架与 AI 助手工作方式，为下一步接入真实业务系统做骨架准备。


## 1.3 三重业务驱动力


### 驱动力 1：一线能力标准化


拜访质量差异是销售团队放大时最常见的问题。系统需要把优秀销售的准备动作、提问套路、价值表达方式和收官方式沉淀下来，让新人也能在短时间内获得可执行的高质量指导。


### 驱动力 2：管理动作可视化


管理者不只关心“结果”，还需要提前发现“过程风险”。例如：


- 哪些 S/A 级客户超期未拜访

- 哪些本周任务尚未确认

- 哪些商机集中在中后期但没有配套拜访动作

- 哪些客户长期没有有效推进


系统通过拜访任务、覆盖率、超期阈值和经营洞察，把这些风险前置暴露出来。


### 驱动力 3：AI 从问答走向作战辅助


单纯的聊天机器人对业务帮助有限。真正有价值的是让 AI 结合客户、任务、商机、知识和方法论一起工作，输出“带上下文的建议”，而不是通用回答。当前系统已经初步形成这一方向：点击客户、任务或知识卡片后，AI 能结合上下文生成对应内容。


## 1.4 价值与 ROI 粗估


由于当前版本仍基于 mock 数据和原型流程，暂不适合给出严格财务口径 ROI，但可以从三个维度说明价值：


| 价值层 | 机制 | 预期收益 |

|------|------|----------|

| 过程提效 | AI 代替销售完成一部分拜访准备、话术草拟、案例匹配和频率检查 | 减少准备时间，提高单人可管客户数 |

| 风险前置 | 用超期、覆盖率、待确认、阶段分布暴露管理盲区 | 降低关键客户漏访与商机停滞风险 |

| 能力复制 | POCC、BPIDC、N-SABE、LSCPA 等方法论嵌入系统 | 缩短新人培养周期，提高打法一致性 |


对于拓斯达这类以项目型销售和深度客户经营为主的组织，系统的第一阶段价值不在“自动成交”，而在“把准备质量、过程纪律和经营可视性做起来”。只要能减少关键客户失控、提高高等级客户覆盖率、缩短新人上手时间，系统就具备较高经营价值。

# 二、产品设计与技术架构


## 2.1 总体架构蓝图（当前代码对应）


当前系统采用“前端工作台 + 本地 Agent 服务 + 可插拔模型/工具层”的结构：


| 层级 | 当前实现 | 说明 |

|------|----------|------|

| 展示层 | React + Vite + Zustand | 承载客户盘点、拜访推进、拜访赋能、经营复盘四大页面 |

| 交互层 | 聊天区 + 卡片点击触发 + 快捷动作 | 用户既可问答，也可通过业务对象触发上下文 |

| 业务规则层 | 本地 fallback 规则引擎 | Agent 不可用时自动兜底，保证演示与基本使用连续性 |

| Agent 运行层 | Node 本地服务 | 对接 OpenAI/兼容接口/Anthropic，并暴露聊天与连通性测试接口 |

| 工具层 | 本地 skill + 远端 MCP 预留 | 已定义多个业务工具，支持后续接外部系统 |

| 数据层 | 当前为 mock 数据 | 后续应替换为 CRM、任务、知识库和经营数据源 |


## 2.2 四大业务工作台


### 2.2.1 客户盘点


客户盘点页当前完成了客户清单底盘，包括：


- 客户名称、等级、行业、地区、商机金额、联系人

- 基于历史拜访计算的超期状态

- SABC 分层规则卡片

- 搜索与等级筛选

- 点击客户后触发 AI 上下文分析


这部分解决的是“底盘可见”的问题，让销售知道自己手上到底有什么客户、哪些客户最该优先经营。


### 2.2.2 拜访推进


拜访推进是当前版本最完整的模块，包含：


- 本周拜访任务

- 待确认拜访提醒

- 任务详情与历史拜访查看

- 已完成拜访

- 拜访覆盖率

- 右侧 AI 拜访助手


这部分本质上是在做一线销售的“每日作战面板”。


### 2.2.3 拜访赋能


拜访赋能页通过知识库承载行业方案、拜访话术、异议处理、竞品对比、成功案例等内容，并按角色层级进行内容过滤，支持：


- 搜索

- 分类筛选

- 热门内容推荐

- 点击知识卡片触发 AI 展开


这部分解决的是“该怎么说、怎么准备、怎么讲案例”的问题。


### 2.2.4 经营复盘


经营复盘页当前实现了面向管理者的多类指标展示：


- 客户总数、商机总额、本周任务、已完成拜访、覆盖率

- 超期客户及涉及商机金额

- 商机阶段分布

- 等级维度商机金额分布

- 商机金额 TOP 客户

- 覆盖率分层明细


这部分为后续 BI 驾驶舱和团队经营分析奠定了视觉与口径基础。


## 2.3 双引擎 AI 机制


当前系统的 AI 能力并不是单一路径，而是采用了“双引擎”设计：


### 引擎 1：本地规则引擎


系统内置了大量业务规则回复与模板生成逻辑，包括但不限于：


- 客户分层规则

- 拜访频率检查

- 覆盖率分析

- 行业案例匹配

- 话术推荐

- 价格异议应对

- BPIDC 提问链

- N-SABE 价值呈现

- C 级客户激活

- POCC 拜访计划


其意义在于：即使外部模型未配置，系统也能完整演示核心业务能力。


### 引擎 2：外部模型 Agent


当配置模型后，前端会优先把当前业务上下文发送给本地 Agent 服务，由 Agent 服务调用外部模型生成回答。当前已支持的接入形态包括：


- OpenAI Responses 或兼容接口

- OpenAI 兼容 `chat/completions`

- Anthropic `messages`


系统同时提供连通性测试与调试信息回显，便于模型接入排障。


### 双引擎价值


双引擎的设计保证了三件事：


1. 没有模型时，系统仍可演示核心价值。

2. 有模型时，系统可以升级为更自然、更强泛化的对话式助手。

3. 未来接入真实工具后，可从“模板型助手”逐步升级为“可执行 Agent”。


## 2.4 当前已定义的业务 Skill


当前服务端已定义 6 个核心业务工具：


| Skill | 功能说明 | 当前状态 |

|------|----------|----------|

| `skill_visit_board_summary` | 拜访看板摘要、任务、完成情况、覆盖率 | 已定义 |

| `skill_customer_snapshot` | 客户画像、联系人、商机、待办与历史拜访 | 已定义 |

| `skill_visit_frequency` | SABC 规则下的频率与超期分析 | 已定义 |

| `skill_industry_cases` | 行业案例与标杆案例匹配 | 已定义 |

| `skill_knowledge_lookup` | 知识库检索、话术与内容展开 | 已定义 |

| `skill_pocc_visit_prep` | 基于客户上下文生成 POCC 拜访准备 | 已定义 |


这些 Skill 目前仍主要基于本地上下文和 mock 数据工作，但工具边界已经成型，方便后续直接替换为真实查询与真实写入动作。


## 2.5 拜访教练模式


系统中还预留了一层更偏“教练化”的增强逻辑，用于处理“拜访准备”“角色扮演”“复盘总结”“快速版指导”等典型场景。其核心思路包括：


- 自动识别用户是新人、资深销售还是管理者

- 自动识别当前诉求是完整准备、快速简报、即时速查、访后复盘还是团队演练

- 根据上下文完整度判断是否需要补采关键信息

- 强制把输出组织到 POCC、BPIDC、N-SABE、LSCPA 等框架中


这意味着系统未来不只是“能答”，而是能按角色和场景切换辅导方式。


## 2.6 千人千面的角色体系


当前前端已经实现分角色视图能力。系统内置了销售总监、区域总监、一线销售等角色视角，不同角色在以下方面存在差异：


| 维度 | 差异内容 |

|------|----------|

| 客户范围 | 一线销售只看自己负责客户；高层可看全局 |

| 知识内容 | 高层偏战略，管理层偏运营，一线偏执行 |

| 问候与上下文 | AI 会结合当前角色生成不同提示 |

| 经营指标 | 当前数据过滤已按角色归属处理 |


这一设计非常关键，因为经营系统不能让所有人看到完全相同的信息，否则会失去实战价值。

# 三、阶段建设成果


## 3.1 客户盘点模块已形成经营底盘


当前客户盘点模块的建设成果主要体现在以下四点：


第一，客户清单已具备经营字段，而不是静态通讯录。系统中每个客户都带有等级、行业、地区、关键联系人、当前商机、金额与阶段等信息。


第二，客户盘点已绑定拜访状态。通过历史拜访记录和 SABC 阈值，系统可自动判断客户是合规、超期还是严重超期。


第三，客户分层规则可视化。系统用规则卡片明确呈现 S/A/B/C 四类客户应有的投入方式和超期阈值。


第四，客户清单可直达 AI 分析。点击任意客户即可进入右侧智能助手，让客户盘点不再停留在看列表，而是直接进入经营动作。


## 3.2 拜访推进模块已形成一线作战面板


当前系统最完整、最适合演示的部分是拜访推进模块，已具备以下能力：


### 3.2.1 本周任务管理


系统按天聚合本周任务，并展示：


- 客户等级

- 拜访目的

- 任务确认状态

- 距上次拜访天数

- 是否超期

- 当前推荐拜访原因


### 3.2.2 待确认提醒


系统支持对待确认任务进行单独提醒，并区分：


- 商机进行中任务

- 无商机但建议识别机会的任务


这使系统从“展示任务”迈向“推动任务确认”。


### 3.2.3 历史拜访回顾


在任务与客户详情中，用户可查看历史拜访时间线，包括：


- 拜访日期

- 拜访摘要

- 结果

- 距上次拜访时间


这对一线销售临出门前快速回顾客户背景非常有帮助。


### 3.2.4 覆盖率管理


系统按 SABC 等级分别计算拜访覆盖情况，并设置预警阈值：


- S 级覆盖率不足时红色预警

- A 级覆盖率不足时提醒

- 高等级未覆盖客户可直接触发 AI


### 3.2.5 访后评估雏形


系统中已经预留并实现了部分访后评估体验，包括基于等级规则的“五个一/三个一”思路，为后续真正形成“有效拜访评估体系”打下基础。


## 3.3 拜访赋能模块已形成知识与方法论入口


当前知识库不是普通文章列表，而是围绕拜访场景组织的知识底盘，已具备：


- 分类化内容组织

- 热门知识推荐

- 按角色过滤知识可见范围

- 与 AI 对话联动展开


从建设意义看，这一步完成的是“把方法论和内容放到系统里”，是从个人经验向组织知识资产迁移的第一步。


## 3.4 经营复盘模块已形成管理视图雏形


当前经营复盘模块虽然仍基于 mock 数据，但已经具备典型经营驾驶舱应有的几个基础视角：


- 经营全景 KPI

- 超期客户风险

- 商机阶段结构

- 等级投入结构

- Top 客户关注

- 覆盖率分析


这意味着系统未来往 BI 层扩展时，不必从零开始设计指标结构。


## 3.5 AI 智能助手已形成可演示闭环


当前 AI 助手的价值不在“万能问答”，而在于已经形成“业务上下文 + 方法论 + 输出模板”的闭环。系统可实现：


- 默认欢迎引导

- 快捷动作触发

- 点击客户后生成客户概况或拜访准备

- 输入业务关键词后生成对应分析

- 外部 Agent 不可用时自动 fallback


从产品成熟度看，这已经不是单个聊天窗口，而是一套嵌入业务工作的 AI 交互层。


## 3.6 最近迭代重点


从最近代码提交可看出，本阶段系统迭代重点集中在一线使用体验优化，主要包括：


- 默认角色改为一线销售

- 任务卡片信息结构优化

- 待确认提醒文案与信息布局调整

- 联系人展示顺序优化

- 详情弹窗重构

- 修复若干渲染崩溃问题


这表明系统正在从“概念原型”向“更贴近日常使用的演示版”收敛。

# 四、数据与知识体系


## 4.1 当前业务数据对象


当前前端数据模型已经覆盖了拜访经营所需的核心对象：


| 对象 | 当前字段重点 |

|------|--------------|

| Customer | 等级、行业、地区、关键联系人、商机、阶段、金额 |

| VisitTask | 任务时间、拜访目的、联系人、状态、上次拜访、预期承诺 |

| CompletedVisit | 拜访摘要、结果、下一步 |

| CoverageData | 覆盖数、总数、按等级分布、未覆盖重点客户 |

| KnowledgeItem | 分类、内容、行业适用、等级适用、标签、热度 |


这套数据结构已经能支撑原型阶段的大部分展示和 AI 分析逻辑。


## 4.2 SABC 分层规则


SABC 是当前系统最重要的业务规则底座之一：


| 等级 | 定义 | 月上门要求 | 月联系要求 | 超期阈值 |

|------|------|------------|------------|----------|

| S | 战略客户 | 4 次 | 4 次 | 7 天 |

| A | 重点客户 | 1 次 | 1 次 | 7 天 |

| B | 活跃客户 | 0 次 | 1 次 | 30 天 |

| C | 沉睡客户 | 0 次 | 1 次 | 30 天 |


系统围绕这套规则完成了以下联动：


- 客户盘点中的拜访状态判定

- 任务中的超期识别

- 覆盖率分层展示

- 有效拜访评估项设计

- AI 中的拜访频率分析和行动建议


## 4.3 行业案例与话术知识


当前系统已配置行业案例与话术规则两类知识底座。


### 行业案例


案例覆盖面板制造、3C 代工、消费电子、家电制造、锂电池等行业，包含：


- 客户名称

- 行业

- 产品线

- 项目金额

- 结果或当前进展


### 话术规则


当前系统已将“客户关注点 × 商机阶段”映射为话术类型和方法论框架，例如：


- BPIDC 提问链

- N-SABE 价值呈现

- STAR-R 成功案例

- LSCPA 异议处理


这类设计非常适合后续演进为“知识检索 + 模板生成 + 动态填充”的组合式能力。


## 4.4 知识分层机制


当前知识库按受众进行过滤：


- 总监：更偏战略与管理内容

- 区域管理者：更偏管理与复盘内容

- 一线销售：更偏执行与话术内容


这意味着知识库的设计逻辑不是“内容越多越好”，而是“角色适配优先”。

# 五、系统边界与现状评估


## 5.1 当前已实现


结合代码现状，可明确判断以下内容已实现：


- 四大主导航页面与主要交互框架

- 一线销售视角下的客户、任务、覆盖率、复盘展示

- 基于 mock 数据的客户、任务、拜访与知识结构

- 右侧 AI 对话区域

- 本地规则引擎与多类业务回复模板

- 本地 Agent 服务与模型接入能力

- OpenAI/兼容接口/Anthropic 的运行时配置能力

- MCP 远端适配层框架


## 5.2 当前已预留但未完全打通


以下能力已经具备结构或入口，但尚未形成真正可上线闭环：


- 远端 MCP 工具真实调用

- 知识库检索式能力

- 真实 CRM 数据查询

- 真实拜访记录写入

- 任务审批或写操作闭环

- 教练模式依赖的外部 Skill 文件完整协同


## 5.3 当前待建设


如果以“可用于真实业务试点”为目标，至少还需要补齐以下能力：


### 数据层


- 接 CRM 客户主数据

- 接真实拜访记录

- 接商机和阶段流转数据

- 接组织架构和权限数据


### Agent 层


- 真实工具读写

- 上下文裁剪与多轮记忆策略

- 知识检索

- 评估与质量校验


### 流程层


- 拜访任务创建/确认/完成闭环

- 访后纪要结构化写入

- 风险信号沉淀

- 管理协同与审批动作


## 5.4 风险与注意事项


### 风险 1：当前仍是 mock 数据驱动


系统展示和 AI 输出当前主要依赖本地静态数据。若直接用于真实业务演示，需要明确说明“这是经营框架原型，不是生产数据系统”。


### 风险 2：拜访教练外部 Skill 路径写死


当前服务端引用的教练 Skill 文件路径是绝对路径，若换机器或换目录，相关增强逻辑可能不可用。


### 风险 3：MCP 适配层已预留，但并非所有协议都已完整打通


尤其是 SSE 风格调用执行仍处于预留阶段，后续需要根据真实外部系统协议补齐。


### 风险 4：当前没有真正写入型业务动作


系统适合做“分析与建议”，但还不能直接承担“创建正式业务记录”的闭环责任。

# 六、下一阶段建设建议


## 6.1 第一阶段：把原型变成可试点系统


建议优先完成以下事项：


1. 用真实 CRM 数据替换 mock 客户、商机和拜访记录。

2. 把“查看类 Skill”接到真实查询接口。

3. 把“拜访记录写入”做成明确审批或确认后的动作。

4. 补齐模型接入配置模板和环境部署说明。


这一阶段的目标不是追求功能全面，而是形成一套能在 1-2 个销售小组中试点运行的闭环。


## 6.2 第二阶段：把 AI 从助手升级为经营 Agent


当数据层打通后，建议把 AI 能力升级到以下方向：


- 自动识别超期客户并生成待办

- 自动生成拜访准备包

- 自动汇总访后纪要

- 自动识别风险信号

- 自动为管理者生成周报摘要


这一阶段的重点是让 AI 不只是“被动响应”，而是开始“主动发现 + 主动建议”。


## 6.3 第三阶段：把复盘做成管理驾驶舱


经营复盘模块可继续扩展到：


- 团队级覆盖率

- 区域级投入产出

- 关键客户风险雷达

- 拜访有效性趋势

- 商机推进与拜访动作关联分析


一旦完成这一步，系统将从“一线作战台”演进为“经营指挥台”。


## 6.4 第四阶段：形成知识与案例自学习闭环


后续建议补齐：


- 访后案例自动沉淀

- 成功/失败经验结构化标签

- 话术效果反馈

- 行业案例评分与排序

- 知识与拜访结果的关联分析


这样系统才能真正沉淀组织资产，而不是持续依赖人工整理。


## 6.5 第五阶段：安全、权限与合规


如果面向真实业务使用，必须同时建设：


- 角色权限控制

- 操作留痕

- 模型输出审计

- 写操作审批

- 数据脱敏与知识边界控制


特别是 AI 输出涉及对外话术、竞品判断和经营建议时，必须有可追溯机制。

# 七、建议演示路径


基于当前系统现状，推荐按以下顺序进行现场演示：


## 7.1 开场：先讲管理问题，不先讲技术


建议从三个问题切入：


- 一线销售拜访准备靠经验，标准不统一

- S/A 级客户有没有真正被高频经营，管理层看不清

- 拜访做了很多，但是否有效、是否推进商机，缺少量化抓手


## 7.2 第一段：客户盘点


展示客户分层、客户列表、超期状态与商机分布，强调这是经营底盘。


## 7.3 第二段：拜访推进


展示本周任务、待确认提醒、覆盖率和历史拜访，强调这是每日作战台。


## 7.4 第三段：AI 助手


通过点击客户或输入“拜访准备”“价格异议”“行业案例”“拜访频率”等关键词，展示系统如何生成具体建议，强调 AI 是业务上下文驱动，而不是通用问答。


## 7.5 第四段：知识赋能


切换角色展示不同层级看到的知识库内容，强调千人千面和组织经验沉淀。


## 7.6 第五段：经营复盘


展示超期客户、阶段分布、Top 客户和覆盖率等管理视角，强调系统对管理者的价值。


## 7.7 收尾：明确“这不是终点，而是骨架”


最后要清楚说明：


- 当前版本已把框架、页面、规则、AI 交互方式与后续扩展结构跑通

- 下一步重点是接真实数据、接真实工具、接真实流程

- 系统最重要的价值在于把拓斯达销售拜访经验沉淀成组织能力

# 八、结论


从当前代码状态看，TopStar 拜访 AI 经营系统已经完成了从“概念想法”到“可演示原型骨架”的关键跨越。


它的核心价值不只是做了几个页面，而是已经把以下几件事连接起来：


- 客户分层规则

- 拜访任务与覆盖率管理

- 经营复盘视图

- 知识赋能

- AI 助手

- 模型接入与工具扩展预留


也就是说，系统已经具备“经营系统雏形”的基本条件。


如果下一阶段能顺利接入真实 CRM、真实拜访记录、真实知识库和真实工具，那么这套系统有机会从“智能拜访助手”进一步升级为“销售经营助手”，最终成为拓斯达销售组织的数字化作战底座。


一句话总结：


不是把销售经验写成 PPT，而是把销售经验做进系统；不是让 AI 只会回答问题，而是让 AI 参与日常经营动作。
